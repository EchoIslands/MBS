# 微信渠道接入规划与指南

> 本文档集中记录 MBS 项目在微信公众号 H5、微信小程序、网页端三端打通过程中的规划、流程、踩坑和经验。
> 任何与微信生态接入相关的问题、方案、教训都应补充到这里，避免散落在各次会话中。

---

## 1. 总体目标

让顾客无论从哪个入口进入 MBS，看到的都是同一个账户、同一份订单、同一张会员卡：

- **网页端**：现有 H5 网页，顾客通过浏览器访问。
- **公众号 H5**：店铺方申请认证服务号 + 已备案域名，将现有 H5 移植到公众号内使用。
- **微信小程序**：后续开发小程序版本，与 H5/网页端共用同一套后端。

所有数据打通，包括：客户数据、预约数据、订单/结算数据、支付数据、会员/积分数据、员工数据。

---

## 2. 当前选定的推进路径

1. **先做 H5**：当前网页端继续迭代，保持可用。
2. **店铺申请域名并备案**：以理发店营业执照为主体购买域名，完成 ICP 备案。
3. **认证微信服务号**：以理发店主体申请并认证服务号（不是订阅号）。
4. **将 H5 移植到店铺域名下**：最终用店铺的网站承载公众号 H5。
5. **再做小程序**：服务号 H5 跑通后，开发小程序版本，与 H5 共用后端和数据。

---

## 2.1 替代路径：小程序优先支付方案

> 如果希望前期尽快跑通支付、降低公众号 H5 支付的复杂度，可以选择**只在小程序内完成支付**，H5/网页端暂时只做预约展示、信息浏览等不涉及资金流转的功能。

### 2.1.1 策略说明

- **H5/网页端**：保留现有功能（预约、排队、查看套餐、会员信息、评价等），但**不接入微信支付**。如需付费，页面引导用户“去小程序完成支付”。
- **小程序**：作为**唯一支付入口**，顾客在小程序内登录、预约、结算、支付。
- **数据打通**：H5 和小程序仍然共用同一套后端和数据库，用户身份通过 unionid 关联，预约订单、会员积分、储值余额等数据完全一致。

### 2.1.2 适用场景

- 想先验证小程序闭环，避免一次性投入公众号认证、H5 支付域名审核等较长周期。
- H5 主要作为信息展示/引流入口，真正的交易闭环放在小程序。
- 店铺暂时不想立刻做 ICP 备案或认证服务号。

### 2.1.3 该路径下的流程变化

| 原路径步骤 | 变化说明 |
|-----------|---------|
| 步骤 1：域名备案 | **可延后**。H5 不做支付时，若 H5 不挂在公众号内，可暂时不备案域名；但若要接入公众号菜单，仍需备案域名。 |
| 步骤 2：微信支付商户号 | **仍然需要**。小程序支付必须绑定商户号。 |
| 步骤 3：微信开放平台 | **仍然需要**。H5 与小程序要打通 unionid，必须绑定同一开放平台。 |
| 步骤 4：认证服务号 | **可延后/可选**。如果 H5 只是普通网页访问，不需要服务号；如果需要公众号菜单入口、模板消息，仍需认证服务号。 |
| 步骤 5-10：公众号 H5 支付相关 | **全部跳过**。不做公众号 H5 支付，不需要 JSAPI、网页授权支付域名等配置。 |
| 步骤 11-17：小程序 | **仍然需要**。小程序认证、商户号绑定、开放平台绑定、服务器域名、登录、支付全部要完成。 |
| 步骤 18-19：数据打通 | **仍然需要**。通过 unionid 把 H5 用户和小程序用户关联起来。 |
| 步骤 20-21：公众号模板消息 | **可延后/可选**。如果暂时不依赖公众号触达用户，可以不做；小程序本身也可以发订阅消息。 |
| 步骤 22：域名切换与 H5 支付配置 | **可大幅简化**。只需保证 H5 域名能正常访问，不需要配置 H5 支付域名。 |

### 2.1.4 该路径的优缺点

**优点：**
- 前期认证、备案、审核环节更少，能更快上线支付闭环。
- 小程序支付体验更稳定，微信原生支持更好。
- H5 可以先作为宣传/预约入口，不承担支付合规压力。

**缺点：**
- 顾客在 H5 看到价格后必须跳转到小程序支付，路径多一步，转化率可能略低。
- 若后续要让 H5 也支持支付，仍需补齐公众号认证、域名备案、H5 支付域名审核等步骤。
- 如果不用服务号，H5 无法获取微信用户信息（unionid 可能为空），跨端身份关联会弱一些。

### 2.1.5 推荐过渡方案

即使选择“只在小程序支付”，也建议按以下顺序推进，避免后期返工：

1. **先申请微信开放平台并完成认证**（数据打通的核心）。
2. **申请微信支付商户号**（小程序支付必需）。
3. **开发小程序并完成支付闭环**（预约 → 结算 → 支付 → 查看订单）。
4. **H5 保留现有功能，增加“去小程序支付”引导**。
5. **后续根据业务需要，再决定是否补齐公众号 H5 支付**。

## 4. 核心机制说明

### 4.1 为什么必须绑定开放平台？

公众号和小程序各自有独立的 openid，同一用户在不同端的 openid 完全不同。只有将两个应用绑定到同一个微信开放平台账号下，才能获取统一的 unionid，这是三端数据打通的基础。

### 4.2 数据打通后的典型场景

1. **跨端登录**：用户在公众号注册后，小程序用同一微信登录，无需重复注册。
2. **跨端支付**：H5 支付、公众号支付、小程序支付都走同一个商户号，资金统一结算。
3. **消息推送**：从小程序触发，向用户在公众号侧发送模板消息（如预约提醒）。

---

## 5. 关键提醒

1. **域名备案主体必须与商户号主体一致**。不一致需提供授权函，且仅限同集团、品牌与经销商、SaaS 服务三种关系。
2. **一个公众号/小程序只能绑定一个开放平台账号**，绑定前确认账号正确。
3. **公众号必须是“已认证的服务号”**，订阅号无网页授权和模板消息权限。
4. **商户号绑定 AppID 后暂不支持解绑**，绑定前确认信息无误。
5. **用户必须关注公众号才能收到模板消息**，需在引导流程中设计关注环节。

---

## 6. 与当前 MBS 系统的关联

- **后端 API**：三端共用现有的 `/api` 路由。
- **数据库**：在 `customers` 表中新增 `openid_oa`、`openid_mini`、`unionid` 字段，用于识别同一用户在不同端的身份。
- **支付**：当前已有的微信支付逻辑（JSAPI）需要适配公众号 H5 和小程序的调用方式。
- **登录**：当前顾客端基于 phone/customerId 的登录需要扩展为支持微信授权登录，但保留原有登录方式作为兜底。

## 8. 决策分析：公众号 H5 支付是否必须做？

> 本节记录 2026-07-16 关于“是否值得投入公众号 H5 支付”的深度分析，供后续决策参考。

### 8.1 基本判断

**结论：公众号 H5 支付不是必须的。对 MBS 当前阶段，优先做小程序支付、公众号只做宣传和预约入口，是更节约资源的合理选择。**

### 8.2 公众号 H5 支付的优劣势

#### 优势

| 优势 | 说明 |
|------|------|
| 用户路径短 | 用户在公众号文章/菜单里直接完成支付，不需要跳转小程序 |
| 营销传播强 | 朋友圈文案、公众号推文可直接挂 H5 支付链接，适合做套餐预售/限时活动 |
| 老用户习惯 | 部分顾客更习惯在公众号内完成操作 |
| 模板消息触达 | 服务号可向已关注用户推送预约提醒、服务完成通知等（触达率通常高于小程序订阅消息） |

#### 劣势

| 劣势 | 说明 |
|------|------|
| 资质成本高 | 必须认证服务号（300 元/年）、域名 ICP 备案、H5 支付域名审核（3-5 天） |
| 开发复杂度高 | 需要网页授权获取 openid、配置 JS 接口安全域名、处理 H5 支付专用接口 |
| 风控更严 | H5 支付要求用户主动点击调起，自动触发容易被微信拦截或降权 |
| 与小程序支付技术栈不同 | 一个是 `WeixinJSBridge.invoke`，一个是 `wx.requestPayment`，需要分别对接和测试 |

### 8.3 小程序支付的优劣势

#### 优势

- 体验更原生，微信官方主推，支付流程稳定。
- 开发相对简单，接口统一。
- 不需要 H5 支付域名审核。
- 天然适合作为交易闭环主入口。

#### 劣势

- 需要单独开发小程序界面。
- 用户无法从朋友圈直接打开小程序，传播路径受限（需扫码或聊天窗口进入）。
- 小程序审核需要周期。

### 8.4 几个容易忽略的关键点

#### （1）用户身份打通会变弱

小程序能拿到 `unionid`，公众号 H5 也能拿到 `unionid`（通过服务号网页授权）。但如果公众号只做宣传平台、不挂需要网页授权的 H5 页面，那么 H5 用户如果没有登录手机号，系统就无法识别他是谁。

**应对方案：**
- H5 端强制用手机号登录；
- 小程序端微信授权后也要绑定手机号；
- 通过手机号把 H5 用户和小程序用户关联起来。

#### （2）公众号的模板消息能力不应浪费

服务号的模板消息（预约提醒、服务完成通知、到账通知）触达效果通常好于小程序订阅消息。如果完全放弃服务号能力，这些触达渠道就没了。

**应对方案：**
- 即使不做 H5 支付，也可以先认证服务号；
- 把服务号作为“消息触达+预约入口”，支付时仍跳小程序。

#### （3）H5 支付的独特价值在朋友圈传播

如果未来做“团购套餐”“限时秒杀”等活动，H5 支付链接可以直接发朋友圈，小程序做不到这一点。这是公众号 H5 支付最大的差异化价值。

#### （4）公众号 H5 支付的真正成本不在开发

H5 已经写好的前提下，接入公众号支付的开发工作量其实不大。真正的成本是：
- 服务号认证 300 元/年
- 域名备案 1-20 个工作日
- H5 支付域名审核 3-5 个工作日
- 网页授权、JS 接口配置等资质类工作

### 8.5 对 MBS 的阶段性建议

#### 第一阶段（当前到上线）：小程序优先

1. 申请微信开放平台并完成认证（unionid 是数据打通核心）。
2. 申请微信支付商户号（小程序支付必需）。
3. 开发小程序，完成预约 → 结算 → 支付 → 查看订单的完整闭环。
4. H5/网页端保留非支付功能（预约、排队、查看、评价、会员查询）。
5. H5 页面在需要支付的位置增加“去小程序完成支付”的引导。

#### 第二阶段（业务稳定后）：按需补齐公众号 H5 支付

当出现以下情况时，再考虑投入公众号 H5 支付：
- H5 跳转小程序那一步导致明显订单流失；
- 需要开展朋友圈营销活动，H5 直接支付更方便；
- 需要服务号模板消息做精准触达，且愿意承担认证和备案成本。

### 8.6 最终结论

- **公众号 H5 支付有价值，但不是 MBS 当前阶段的必选项。**
- **小程序支付优先 + 公众号做宣传/预约入口**，既能尽快跑通交易闭环，又能节约前期资质和开发成本。
- **唯一需要提前规划的是用户身份打通**：即使不做 H5 支付，也要让 H5 用户留下手机号，否则跨端数据无法合并。

## 10. 开发小程序前的技术准备

> 小程序不是独立 App，必须与现有 H5/网页端共用同一套后端和数据。本节分析在写小程序代码之前，应该完成哪些前期准备，才能保证最终三端数据相通、体验一致。

### 10.1 核心原则

- **后端一套，多端共用**：小程序、H5、网页端调用同一组 `/api` 接口。
- **数据一份，统一索引**：顾客、预约、订单、支付等数据都存在同一个 Supabase 数据库。
- **身份统一，登录态隔离**：不同端用不同的登录方式获取 token，但识别的是同一个顾客。

---

### 10.2 数据层准备

#### 10.2.1 统一类型定义

- 检查 `/shared/types.ts` 是否完整覆盖：
  - 顾客（Customer）
  - 预约（Booking）
  - 结算/订单（Settlement / Order）
  - 支付（Payment）
  - 服务/商品（Service / Product）
  - 员工（Employee）
  - 店铺（Shop）
- 小程序端直接复用这些类型，避免两端各自定义导致字段不一致。

#### 10.2.2 数据库字段扩展

至少需要为顾客表增加微信生态字段：

```sql
alter table customers add column if not exists openid_oa text;
alter table customers add column if not exists openid_mini text;
alter table customers add column if not exists unionid text;
alter table customers add column if not exists wechat_avatar text;
alter table customers add column if not exists wechat_nickname text;
```

**关键点：**
- `unionid` 是跨端身份打通的核心；
- `openid_mini` 用于小程序支付和推送；
- `openid_oa` 用于公众号模板消息（如果后续做）。

#### 10.2.3 顾客身份合并策略

必须提前确定：小程序用户进来后，如何与现有顾客记录关联？

**推荐方案：手机号优先匹配**

1. 小程序端 `wx.login()` 拿到 code；
2. 后端换取 `openid` 和 `unionid`；
3. 小程序端引导用户授权手机号（`getPhoneNumber`）；
4. 后端用手机号在 `customers` 表中查找：
   - 找到：把 `openid_mini` 和 `unionid` 更新到该记录；
   - 未找到：用手机号新建顾客记录，并写入 `openid_mini` 和 `unionid`。

**备选方案：unionid 优先匹配**

- 如果 H5 已经通过公众号网页授权拿到了 `unionid`，可以直接用 `unionid` 匹配。
- 但当前 H5 还没接入公众号授权，所以**手机号优先更现实**。

### 10.3 API 层准备

#### 10.3.1 接口返回格式统一

当前后端接口返回格式：

```json
{ "success": true, "data": { ... } }
```

小程序端直接复用这套格式，不要另起一套。

#### 10.3.2 鉴权体系梳理

当前系统里有几类用户：

| 用户类型 | 当前鉴权方式 | 是否支持小程序 |
|----------|-------------|---------------|
| 员工 | JWT token（`authMiddleware`） | 否 |
| H5 顾客 | `customerId` 存在本地 storage | 否，不安全 |
| 小程序顾客 | 需要新增 | 待开发 |

**必须新增：顾客端统一鉴权**

建议引入自定义顾客 token（也可以用 JWT）：

```
POST /api/auth/customer/login
请求：{ phone, code? }  // H5 用手机号+验证码，小程序用 code
响应：{ success: true, data: { token, customer } }
```

然后新增一个 `customerAuthMiddleware`，校验顾客 token：

```ts
// 保护顾客端敏感接口
/customer/profile
/customer/bookings
/customer/payments
```

**关键点：**
- 员工接口和顾客接口要分开保护；
- 小程序登录和 H5 手机号登录最终都返回同一个顾客 token；
- 支付、查看订单等接口用顾客 token 保护，不要再传 `customerId` 明文。

#### 10.3.3 小程序专用接口

需要新增（或改造）以下接口：

| 接口 | 说明 |
|------|------|
| `POST /api/auth/mini-login` | 小程序 code 换登录态 |
| `POST /api/auth/mini-phone` | 小程序授权手机号 |
| `POST /api/payments/mini-prepay` | 小程序支付统一下单 |
| `PUT /api/bookings/:id/cancel` | 顾客取消预约（已在 3.15.6 修复） |

---

### 10.4 认证体系设计

#### 10.4.1 推荐方案：双 token 体系

保留现有员工 JWT，新增顾客 JWT：

- 员工 token payload：`{ id, shopId, role, name, phone }`
- 顾客 token payload：`{ customerId, phone, shopId }`

前端存储：
- H5：`localStorage.setItem('customer_token', token)`
- 小程序：`wx.setStorageSync('customer_token', token)`

接口调用：
- 员工接口：`Authorization: Bearer <employee_token>`
- 顾客接口：`Authorization: Bearer <customer_token>`

#### 10.4.2 为什么不能用 `customerId` 明文传参？

当前 H5 很多接口直接传 `customerId`，比如：

```ts
bookingApi.getMyBookings(currentCustomer.id)
```

这在小程序里也能跑，但存在安全问题：用户可以改 `customerId` 查看别人的数据。所以必须改成 token 鉴权。

**改造范围：**
- 所有 `/api/customer/*` 接口；
- 所有涉及顾客隐私的查询（预约、订单、余额、积分）。

### 10.5 支付层准备

#### 10.5.1 当前 H5 支付现状

需要先确认当前 H5 支付是怎么实现的：
- 是否已经是微信支付 JSAPI？
- 支付回调接口在哪里？
- 订单/结算记录如何生成？

#### 10.5.2 抽象支付服务

建议后端新增一个 `paymentService`，统一处理：

```ts
// 创建支付订单（渠道无关）
createPaymentOrder({ bookingId, settlementId, channel: 'wechat_mini' | 'wechat_h5', amount })

// 微信统一下单（根据渠道返回不同调起参数）
unifiedOrder({ paymentId, channel, openid })

// 支付回调（统一更新订单状态）
handleWechatCallback(xmlData)
```

这样小程序和 H5 共用同一套订单和回调逻辑，只有前端调起方式不同。

#### 10.5.3 支付表设计

如果还没有独立的支付记录表，建议新增：

```sql
create table if not exists payments (
  id text primary key,
  customer_id text references customers(id),
  booking_id text references bookings(id),
  settlement_id text references settlements(id),
  channel text, -- wechat_mini / wechat_h5 / cash / balance
  amount numeric,
  status text default 'pending', -- pending / paid / failed / refunded
  transaction_id text, -- 微信支付订单号
  paid_at timestamptz,
  created_at timestamptz default now()
);
```

### 10.6 前端架构选择

#### 10.6.1 方案 A：小程序独立开发（推荐当前阶段）

- H5 继续用 React + Vite；
- 小程序用原生或 Taro/React 开发；
- 两端独立维护，但共用后端 API 和类型定义。

**优点：**
- 技术栈清晰，互不干扰；
- 小程序可以完全按微信规范优化体验；
- H5 改动风险小。

**缺点：**
- 部分 UI 逻辑需要写两遍。

#### 10.6.2 方案 B：跨端框架（Taro / Uni-app）

- 一套代码同时输出 H5 和小程序。

**优点：**
- 减少重复开发。

**缺点：**
- 改造现有 H5 成本大；
- 框架限制多，微信支付、地图等原生能力适配复杂；
- 对当前项目来说，ROI 不高。

**建议：当前用方案 A，业务稳定后再评估是否迁移跨端框架。**

### 10.10 小程序与 H5 独立开发时如何保证数据与逻辑同步

> 小程序和 H5 前端代码各自维护，但数据必须同源、核心逻辑必须复用。本节给出具体可执行的措施。

#### 10.10.1 数据同源（已具备）

- 网页端与 H5 共用同一套 React + Vite 代码，天然同步；
- 小程序单独开发前端，但调用同一组 `/api` 接口和同一个 Supabase 数据库；
- 顾客、预约、订单、积分、余额等核心数据只有一份，不存在多副本同步问题。

#### 10.10.2 复用层必须抽到 `shared/`

| 复用内容 | 当前位置 | 目标位置 | 说明 |
|----------|----------|----------|------|

#### 10.10.3 流程与命名一致

- 核心操作流程两端必须一致：预约 → 确认 → 结算 → 支付 → 评价；
- 状态文案统一：待确认、已确认、已完成、已取消；
- 页面结构统一：首页、预约、排队、个人中心；
- 同一功能在不同端使用相同名称，避免用户混淆。

**建议维护一份"H5/小程序页面映射表"：**

#### 10.10.4 允许差异的地方

- **支付调起**：H5 用 `WeixinJSBridge.invoke`，小程序用 `wx.requestPayment`；
- **登录方式**：H5 用手机号，小程序用 `wx.login` + 手机号授权；
- **原生能力**：小程序可调扫码、订阅消息；H5 需走公众号能力。

这些差异应在架构上隔离：后端逻辑统一，前端只在"最后一公里"按平台适配。
