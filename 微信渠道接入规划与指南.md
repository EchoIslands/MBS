# 微信渠道接入规划与指南

> 本文档集中记录 MBS 项目在微信公众号 H5、微信小程序、网页端三端打通过程中的规划、流程、踩坑和经验。
> 任何与微信生态接入相关的问题、方案、教训都应补充到这里，避免散落在各次会话中。

---

## 1. 总体目标

让顾客无论从哪个入口进入 MBS，看到的都是同一个账户、同一份订单、同一张会员卡：

- **网页端**：现有 H5 网页，顾客通过浏览器访问。
- **公众号 H5**：店铺方申请认证服务号 + 已备案域名，将现有 H5 移植到公众号内使用。
- **微信小程序**：后续开发小程序版本，与 H5/网页端共用同一套后端。

所有数据打通，包括：客户数据、预约数据、订单/结算数据、支付数据、会员/积分数据、员工数据。

---

## 2. 当前选定的推进路径

1. **先做 H5**：当前网页端继续迭代，保持可用。
2. **店铺申请域名并备案**：以理发店营业执照为主体购买域名，完成 ICP 备案。
3. **认证微信服务号**：以理发店主体申请并认证服务号（不是订阅号）。
4. **将 H5 移植到店铺域名下**：最终用店铺的网站承载公众号 H5。
5. **再做小程序**：服务号 H5 跑通后，开发小程序版本，与 H5 共用后端和数据。

---

## 2.1 替代路径：小程序优先支付方案

> 如果希望前期尽快跑通支付、降低公众号 H5 支付的复杂度，可以选择**只在小程序内完成支付**，H5/网页端暂时只做预约展示、信息浏览等不涉及资金流转的功能。

### 2.1.1 策略说明

- **H5/网页端**：保留现有功能（预约、排队、查看套餐、会员信息、评价等），但**不接入微信支付**。如需付费，页面引导用户“去小程序完成支付”。
- **小程序**：作为**唯一支付入口**，顾客在小程序内登录、预约、结算、支付。
- **数据打通**：H5 和小程序仍然共用同一套后端和数据库，用户身份通过 unionid 关联，预约订单、会员积分、储值余额等数据完全一致。

### 2.1.2 适用场景

- 想先验证小程序闭环，避免一次性投入公众号认证、H5 支付域名审核等较长周期。
- H5 主要作为信息展示/引流入口，真正的交易闭环放在小程序。
- 店铺暂时不想立刻做 ICP 备案或认证服务号。

### 2.1.3 该路径下的流程变化

| 原路径步骤 | 变化说明 |
|-----------|---------|
| 步骤 1：域名备案 | **可延后**。H5 不做支付时，若 H5 不挂在公众号内，可暂时不备案域名；但若要接入公众号菜单，仍需备案域名。 |
| 步骤 2：微信支付商户号 | **仍然需要**。小程序支付必须绑定商户号。 |
| 步骤 3：微信开放平台 | **仍然需要**。H5 与小程序要打通 unionid，必须绑定同一开放平台。 |
| 步骤 4：认证服务号 | **可延后/可选**。如果 H5 只是普通网页访问，不需要服务号；如果需要公众号菜单入口、模板消息，仍需认证服务号。 |
| 步骤 5-10：公众号 H5 支付相关 | **全部跳过**。不做公众号 H5 支付，不需要 JSAPI、网页授权支付域名等配置。 |
| 步骤 11-17：小程序 | **仍然需要**。小程序认证、商户号绑定、开放平台绑定、服务器域名、登录、支付全部要完成。 |
| 步骤 18-19：数据打通 | **仍然需要**。通过 unionid 把 H5 用户和小程序用户关联起来。 |
| 步骤 20-21：公众号模板消息 | **可延后/可选**。如果暂时不依赖公众号触达用户，可以不做；小程序本身也可以发订阅消息。 |
| 步骤 22：域名切换与 H5 支付配置 | **可大幅简化**。只需保证 H5 域名能正常访问，不需要配置 H5 支付域名。 |

### 2.1.4 该路径的优缺点

**优点：**
- 前期认证、备案、审核环节更少，能更快上线支付闭环。
- 小程序支付体验更稳定，微信原生支持更好。
- H5 可以先作为宣传/预约入口，不承担支付合规压力。

**缺点：**
- 顾客在 H5 看到价格后必须跳转到小程序支付，路径多一步，转化率可能略低。
- 若后续要让 H5 也支持支付，仍需补齐公众号认证、域名备案、H5 支付域名审核等步骤。
- 如果不用服务号，H5 无法获取微信用户信息（unionid 可能为空），跨端身份关联会弱一些。

### 2.1.5 推荐过渡方案

即使选择“只在小程序支付”，也建议按以下顺序推进，避免后期返工：

1. **先申请微信开放平台并完成认证**（数据打通的核心）。
2. **申请微信支付商户号**（小程序支付必需）。
3. **开发小程序并完成支付闭环**（预约 → 结算 → 支付 → 查看订单）。
4. **H5 保留现有功能，增加“去小程序支付”引导**。
5. **后续根据业务需要，再决定是否补齐公众号 H5 支付**。

## 4. 核心机制说明

### 4.1 为什么必须绑定开放平台？

公众号和小程序各自有独立的 openid，同一用户在不同端的 openid 完全不同。只有将两个应用绑定到同一个微信开放平台账号下，才能获取统一的 unionid，这是三端数据打通的基础。

### 4.2 数据打通后的典型场景

1. **跨端登录**：用户在公众号注册后，小程序用同一微信登录，无需重复注册。
2. **跨端支付**：H5 支付、公众号支付、小程序支付都走同一个商户号，资金统一结算。
3. **消息推送**：从小程序触发，向用户在公众号侧发送模板消息（如预约提醒）。

---

## 5. 关键提醒

1. **域名备案主体必须与商户号主体一致**。不一致需提供授权函，且仅限同集团、品牌与经销商、SaaS 服务三种关系。
2. **一个公众号/小程序只能绑定一个开放平台账号**，绑定前确认账号正确。
3. **公众号必须是“已认证的服务号”**，订阅号无网页授权和模板消息权限。
4. **商户号绑定 AppID 后暂不支持解绑**，绑定前确认信息无误。
5. **用户必须关注公众号才能收到模板消息**，需在引导流程中设计关注环节。

---

## 6. 与当前 MBS 系统的关联

- **后端 API**：三端共用现有的 `/api` 路由。
- **数据库**：在 `customers` 表中新增 `openid_oa`、`openid_mini`、`unionid` 字段，用于识别同一用户在不同端的身份。
- **支付**：当前已有的微信支付逻辑（JSAPI）需要适配公众号 H5 和小程序的调用方式。
- **登录**：当前顾客端基于 phone/customerId 的登录需要扩展为支持微信授权登录，但保留原有登录方式作为兜底。

## 7. 踩坑与经验记录

> 按时间倒序记录实际推进中遇到的问题和解决方案。

### 2026-07-21

- **进度 1：小程序 UI 与 H5 统一改造完成**。按“顾客端三端视觉一致”原则，对小程序首页、预约页、排队页、个人中心、结算页做一次统一改造，确保用户在不同入口看到相同的颜色、字体、按钮、图标、布局和交互习惯。
  - 建立 `mini-program/app.scss` 设计 Token：品牌色、灰阶、语义色、字体、圆角、阴影、按钮、卡片、标签、徽章、渐变头部、底部固定栏等全部统一封装，禁止页面样式硬编码色值；
  - 新增通用图标组件 `mini-program/components/icon/icon`：用 lucide 风格 SVG 复刻 H5 图标，页面统一通过 `<mbs-icon name="xxx" />` 引用，图标名与 H5 lucide-react 图标保持一致；
  - 重构 5 个核心页面：
    - 首页 `pages/index/index`：渐变头部、店铺信息卡片、服务列表、发型师团队、底部预约按钮；
    - 预约页 `pages/booking/booking`：卡片式服务/日期/时间/发型师选择、智能匹配、底部固定结算栏；
    - 排队页 `pages/queue/queue`：服务进度可视化、当前叫号、预计等待、队列列表；
    - 个人中心 `pages/profile/profile`：会员信息卡片、折扣展示、权益对比表格、预约列表、快捷功能、预约详情弹窗；
    - 结算页 `pages/checkout/checkout`：预约信息卡片、会员折扣明细、支付方式说明、底部预计实付栏；
  - 新增 `mini-program/utils/membership.js`：统一会员等级、折扣、有效期、权益计算，供个人中心和结算页复用，避免两端各自维护会员逻辑；
  - 统一登录体验：抽取 `components/login-modal/login-modal`，在“我的”、点击服务、确认预约、结算等需要登录的场景自动弹窗，登录成功后刷新当前页面数据；
  - 修复 `profile.json` 未注册 `mbs-icon` 组件的问题，避免个人中心图标渲染报错。
- **进度 2：自测三轮并修复实际 bug**。按“测试 3 遍，遇到 bug 自行修改”的要求，对小程序做静态检查 + 数据绑定检查 + `npm run build` 验证，发现并修复以下问题：
  - `mini-program/app.scss` 缺少常用 margin / padding / grid / rounded / background 工具类，导致 profile/checkout/queue 中 `mb-4`、`mt-1`、`grid`、`rounded-full`、`bg-white` 等类失效；已在 `app.scss` 统一补齐，并删除 `queue.scss` 中重复定义；
  - `pages/queue/queue.js` 取参使用 `options.id`，与其他页面统一使用的 `bookingId` 不一致，导致 tabBar 跳转后无法加载指定预约；已改为 `options.bookingId || options.id || routeParams.bookingId`；
  - `pages/queue/queue.wxml` 混用 `<span>` 标签和含点的 `gap-1.5` 类，微信小程序不支持；已将 `<span>` 改为 `<text>`，`gap-1.5` 改为 `gap-1_5` 并在 `app.scss` 定义；
  - `pages/booking/booking.js` 未通过 `takeRouteParams()` 读取首页传递的 `serviceId`，导致首页点击服务后预约页没有预选服务；已增加 `onLoad`/`onShow` 双通道读取；
  - `pages/booking/booking.wxml` 时间格渲染把 `timeSlots` 对象直接当字符串用（`{{item}}`、`selectedTime === item`），导致选中状态与“已满”标签全部失效；已改为 `item.time` / `item.full`；
  - `npm run build` 因环境缺少 `node_modules` 先执行 `npm install`，最终 H5 build 通过。
- **经验 1：UI 改动必须同步评估三端**。颜色、字体、图标、状态徽章、按钮样式等任何视觉调整，都会同时影响 H5、小程序，甚至后端返回字段（如会员权益文案）。后续任何界面调整必须：
  1. 以 H5 顾客端为基准确认设计；
  2. 同步改 `mini-program/app.scss` / `components/icon` / 页面样式；
  3. 同步改 `shared/lib/membership.ts` 等共享层；
  4. 两端同时目测回归关键页面（首页、预约、排队、个人中心、结算）。
- **进度 3：根据真机调试反馈与 H5 一致性核对再做调整**。用户在微信开发者工具真机调试时发现：
  - `components/icon/icon.scss` 使用 `.mbs-icon svg` 标签选择器，触发微信小程序组件 wxss 限制警告；已改为 `.mbs-icon-svg` 类选择器，并给所有 `<svg>` 节点补 `class="mbs-icon-svg"`；
  - 首页标题出现“附近理发店”字样，与单店定位冲突；已改为 `{{shop.name || '皓诗形象设计'}}`；

## 8. 决策分析：公众号 H5 支付是否必须做？

> 本节记录 2026-07-16 关于“是否值得投入公众号 H5 支付”的深度分析，供后续决策参考。

### 8.1 基本判断

**结论：公众号 H5 支付不是必须的。对 MBS 当前阶段，优先做小程序支付、公众号只做宣传和预约入口，是更节约资源的合理选择。**

### 8.2 公众号 H5 支付的优劣势

#### 优势

| 优势 | 说明 |
|------|------|
| 用户路径短 | 用户在公众号文章/菜单里直接完成支付，不需要跳转小程序 |
| 营销传播强 | 朋友圈文案、公众号推文可直接挂 H5 支付链接，适合做套餐预售/限时活动 |
| 老用户习惯 | 部分顾客更习惯在公众号内完成操作 |
| 模板消息触达 | 服务号可向已关注用户推送预约提醒、服务完成通知等（触达率通常高于小程序订阅消息） |

#### 劣势

| 劣势 | 说明 |
|------|------|
| 资质成本高 | 必须认证服务号（300 元/年）、域名 ICP 备案、H5 支付域名审核（3-5 天） |
| 开发复杂度高 | 需要网页授权获取 openid、配置 JS 接口安全域名、处理 H5 支付专用接口 |
| 风控更严 | H5 支付要求用户主动点击调起，自动触发容易被微信拦截或降权 |
| 与小程序支付技术栈不同 | 一个是 `WeixinJSBridge.invoke`，一个是 `wx.requestPayment`，需要分别对接和测试 |

### 8.3 小程序支付的优劣势

#### 优势

- 体验更原生，微信官方主推，支付流程稳定。
- 开发相对简单，接口统一。
- 不需要 H5 支付域名审核。
- 天然适合作为交易闭环主入口。

#### 劣势

- 需要单独开发小程序界面。
- 用户无法从朋友圈直接打开小程序，传播路径受限（需扫码或聊天窗口进入）。
- 小程序审核需要周期。

### 8.4 几个容易忽略的关键点

#### （1）用户身份打通会变弱

小程序能拿到 `unionid`，公众号 H5 也能拿到 `unionid`（通过服务号网页授权）。但如果公众号只做宣传平台、不挂需要网页授权的 H5 页面，那么 H5 用户如果没有登录手机号，系统就无法识别他是谁。

**应对方案：**
- H5 端强制用手机号登录；
- 小程序端微信授权后也要绑定手机号；
- 通过手机号把 H5 用户和小程序用户关联起来。

#### （2）公众号的模板消息能力不应浪费

服务号的模板消息（预约提醒、服务完成通知、到账通知）触达效果通常好于小程序订阅消息。如果完全放弃服务号能力，这些触达渠道就没了。

**应对方案：**
- 即使不做 H5 支付，也可以先认证服务号；
- 把服务号作为“消息触达+预约入口”，支付时仍跳小程序。

#### （3）H5 支付的独特价值在朋友圈传播

如果未来做“团购套餐”“限时秒杀”等活动，H5 支付链接可以直接发朋友圈，小程序做不到这一点。这是公众号 H5 支付最大的差异化价值。

#### （4）公众号 H5 支付的真正成本不在开发

H5 已经写好的前提下，接入公众号支付的开发工作量其实不大。真正的成本是：
- 服务号认证 300 元/年
- 域名备案 1-20 个工作日
- H5 支付域名审核 3-5 个工作日
- 网页授权、JS 接口配置等资质类工作

### 8.5 对 MBS 的阶段性建议

#### 第一阶段（当前到上线）：小程序优先

1. 申请微信开放平台并完成认证（unionid 是数据打通核心）。
2. 申请微信支付商户号（小程序支付必需）。
3. 开发小程序，完成预约 → 结算 → 支付 → 查看订单的完整闭环。
4. H5/网页端保留非支付功能（预约、排队、查看、评价、会员查询）。
5. H5 页面在需要支付的位置增加“去小程序完成支付”的引导。

#### 第二阶段（业务稳定后）：按需补齐公众号 H5 支付

当出现以下情况时，再考虑投入公众号 H5 支付：
- H5 跳转小程序那一步导致明显订单流失；
- 需要开展朋友圈营销活动，H5 直接支付更方便；
- 需要服务号模板消息做精准触达，且愿意承担认证和备案成本。

### 8.6 最终结论

- **公众号 H5 支付有价值，但不是 MBS 当前阶段的必选项。**
- **小程序支付优先 + 公众号做宣传/预约入口**，既能尽快跑通交易闭环，又能节约前期资质和开发成本。
- **唯一需要提前规划的是用户身份打通**：即使不做 H5 支付，也要让 H5 用户留下手机号，否则跨端数据无法合并。

## 10. 开发小程序前的技术准备

> 小程序不是独立 App，必须与现有 H5/网页端共用同一套后端和数据。本节分析在写小程序代码之前，应该完成哪些前期准备，才能保证最终三端数据相通、体验一致。

### 10.1 核心原则

- **后端一套，多端共用**：小程序、H5、网页端调用同一组 `/api` 接口。
- **数据一份，统一索引**：顾客、预约、订单、支付等数据都存在同一个 Supabase 数据库。
- **身份统一，登录态隔离**：不同端用不同的登录方式获取 token，但识别的是同一个顾客。

---

### 10.2 数据层准备

#### 10.2.1 统一类型定义

- 检查 `/shared/types.ts` 是否完整覆盖：
  - 顾客（Customer）
  - 预约（Booking）
  - 结算/订单（Settlement / Order）
  - 支付（Payment）
  - 服务/商品（Service / Product）
  - 员工（Employee）
  - 店铺（Shop）
- 小程序端直接复用这些类型，避免两端各自定义导致字段不一致。

#### 10.2.2 数据库字段扩展

至少需要为顾客表增加微信生态字段：

```sql
alter table customers add column if not exists openid_oa text;
alter table customers add column if not exists openid_mini text;
alter table customers add column if not exists unionid text;
alter table customers add column if not exists wechat_avatar text;
alter table customers add column if not exists wechat_nickname text;
```

**关键点：**
- `unionid` 是跨端身份打通的核心；
- `openid_mini` 用于小程序支付和推送；
- `openid_oa` 用于公众号模板消息（如果后续做）。

#### 10.2.3 顾客身份合并策略

必须提前确定：小程序用户进来后，如何与现有顾客记录关联？

**推荐方案：手机号优先匹配**

1. 小程序端 `wx.login()` 拿到 code；
2. 后端换取 `openid` 和 `unionid`；
3. 小程序端引导用户授权手机号（`getPhoneNumber`）；
4. 后端用手机号在 `customers` 表中查找：
   - 找到：把 `openid_mini` 和 `unionid` 更新到该记录；
   - 未找到：用手机号新建顾客记录，并写入 `openid_mini` 和 `unionid`。

**备选方案：unionid 优先匹配**

- 如果 H5 已经通过公众号网页授权拿到了 `unionid`，可以直接用 `unionid` 匹配。
- 但当前 H5 还没接入公众号授权，所以**手机号优先更现实**。

---

### 10.3 API 层准备

#### 10.3.1 接口返回格式统一

当前后端接口返回格式：

```json
{ "success": true, "data": { ... } }
```

小程序端直接复用这套格式，不要另起一套。

#### 10.3.2 鉴权体系梳理

当前系统里有几类用户：

| 用户类型 | 当前鉴权方式 | 是否支持小程序 |
|----------|-------------|---------------|
| 员工 | JWT token（`authMiddleware`） | 否 |
| H5 顾客 | `customerId` 存在本地 storage | 否，不安全 |
| 小程序顾客 | 需要新增 | 待开发 |

**必须新增：顾客端统一鉴权**

建议引入自定义顾客 token（也可以用 JWT）：

```
POST /api/auth/customer/login
请求：{ phone, code? }  // H5 用手机号+验证码，小程序用 code
响应：{ success: true, data: { token, customer } }
```

然后新增一个 `customerAuthMiddleware`，校验顾客 token：

```ts
// 保护顾客端敏感接口
/customer/profile
/customer/bookings
/customer/payments
```

**关键点：**
- 员工接口和顾客接口要分开保护；
- 小程序登录和 H5 手机号登录最终都返回同一个顾客 token；
- 支付、查看订单等接口用顾客 token 保护，不要再传 `customerId` 明文。

#### 10.3.3 小程序专用接口

需要新增（或改造）以下接口：

| 接口 | 说明 |
|------|------|
| `POST /api/auth/mini-login` | 小程序 code 换登录态 |
| `POST /api/auth/mini-phone` | 小程序授权手机号 |
| `POST /api/payments/mini-prepay` | 小程序支付统一下单 |
| `PUT /api/bookings/:id/cancel` | 顾客取消预约（已在 3.15.6 修复） |

---

### 10.4 认证体系设计

#### 10.4.1 推荐方案：双 token 体系

保留现有员工 JWT，新增顾客 JWT：

- 员工 token payload：`{ id, shopId, role, name, phone }`
- 顾客 token payload：`{ customerId, phone, shopId }`

前端存储：
- H5：`localStorage.setItem('customer_token', token)`
- 小程序：`wx.setStorageSync('customer_token', token)`

接口调用：
- 员工接口：`Authorization: Bearer <employee_token>`
- 顾客接口：`Authorization: Bearer <customer_token>`

#### 10.4.2 为什么不能用 `customerId` 明文传参？

当前 H5 很多接口直接传 `customerId`，比如：

```ts
bookingApi.getMyBookings(currentCustomer.id)
```

这在小程序里也能跑，但存在安全问题：用户可以改 `customerId` 查看别人的数据。所以必须改成 token 鉴权。

**改造范围：**
- 所有 `/api/customer/*` 接口；
- 所有涉及顾客隐私的查询（预约、订单、余额、积分）。

---

### 10.5 支付层准备

#### 10.5.1 当前 H5 支付现状

需要先确认当前 H5 支付是怎么实现的：
- 是否已经是微信支付 JSAPI？
- 支付回调接口在哪里？
- 订单/结算记录如何生成？

#### 10.5.2 抽象支付服务

建议后端新增一个 `paymentService`，统一处理：

```ts
// 创建支付订单（渠道无关）
createPaymentOrder({ bookingId, settlementId, channel: 'wechat_mini' | 'wechat_h5', amount })

// 微信统一下单（根据渠道返回不同调起参数）
unifiedOrder({ paymentId, channel, openid })

// 支付回调（统一更新订单状态）
handleWechatCallback(xmlData)
```

这样小程序和 H5 共用同一套订单和回调逻辑，只有前端调起方式不同。

#### 10.5.3 支付表设计

如果还没有独立的支付记录表，建议新增：

```sql
create table if not exists payments (
  id text primary key,
  customer_id text references customers(id),
  booking_id text references bookings(id),
  settlement_id text references settlements(id),
  channel text, -- wechat_mini / wechat_h5 / cash / balance
  amount numeric,
  status text default 'pending', -- pending / paid / failed / refunded
  transaction_id text, -- 微信支付订单号
  paid_at timestamptz,
  created_at timestamptz default now()
);
```

### 10.6 前端架构选择

#### 10.6.1 方案 A：小程序独立开发（推荐当前阶段）

- H5 继续用 React + Vite；
- 小程序用原生或 Taro/React 开发；
- 两端独立维护，但共用后端 API 和类型定义。

**优点：**
- 技术栈清晰，互不干扰；
- 小程序可以完全按微信规范优化体验；
- H5 改动风险小。

**缺点：**
- 部分 UI 逻辑需要写两遍。

#### 10.6.2 方案 B：跨端框架（Taro / Uni-app）

- 一套代码同时输出 H5 和小程序。

**优点：**
- 减少重复开发。

**缺点：**
- 改造现有 H5 成本大；
- 框架限制多，微信支付、地图等原生能力适配复杂；
- 对当前项目来说，ROI 不高。

**建议：当前用方案 A，业务稳定后再评估是否迁移跨端框架。**

### 10.10 小程序与 H5 独立开发时如何保证数据与逻辑同步

> 小程序和 H5 前端代码各自维护，但数据必须同源、核心逻辑必须复用。本节给出具体可执行的措施。

#### 10.10.1 数据同源（已具备）

- 网页端与 H5 共用同一套 React + Vite 代码，天然同步；
- 小程序单独开发前端，但调用同一组 `/api` 接口和同一个 Supabase 数据库；
- 顾客、预约、订单、积分、余额等核心数据只有一份，不存在多副本同步问题。

#### 10.10.2 复用层必须抽到 `shared/`

| 复用内容 | 当前位置 | 目标位置 | 说明 |
|----------|----------|----------|------|
| 实体类型 | `shared/types.ts` | 保持 | Customer、Booking、Settlement、Payment 等 |
| API 请求封装 | `src/api.ts` | `shared/api.ts` | 不依赖浏览器 DOM 的请求逻辑、接口函数签名 |
| 业务工具函数 | `src/lib/` | `shared/lib/` | 会员折扣、积分计算、排队等待时间等纯计算逻辑 |
| Mock 数据 | `shared/mockData.ts` | 保持 | 开发期兜底数据 |

**操作步骤（已完成）：**

1. ✅ 在 `shared/` 下新建 `api-base.ts`，将通用 `http` 封装、API 基础地址获取、真实 API 开关迁移过来；
2. ✅ 在 `shared/lib/` 下新建 `membership.ts` 和 `avatar.ts`，将会员折扣、积分、VIP 有效期、头像 URL 生成等纯函数迁移过来；

4. ✅ `src/lib/membership.ts`、`src/lib/avatar.ts` 改为从 `shared/lib/` 重新导出，保持 H5 现有 import 路径不变。

#### 10.10.3 流程与命名一致

- 核心操作流程两端必须一致：预约 → 确认 → 结算 → 支付 → 评价；
- 状态文案统一：待确认、已确认、已完成、已取消；
- 页面结构统一：首页、预约、排队、个人中心；
- 同一功能在不同端使用相同名称，避免用户混淆。

**建议维护一份"H5/小程序页面映射表"：**

| H5 页面 | 小程序页面 | 说明 |
|---------|-----------|------|
| `/customer/home` | `pages/index/index` | 首页 |
| `/customer/booking/:shopId` | `pages/booking/index` | 预约 |
| `/customer/queue/:id` | `pages/queue/index` | 排队 |
| `/customer/profile` | `pages/profile/index` | 个人中心 |
| `/customer/checkout` | `pages/checkout/index` | 结算 |

#### 10.10.4 允许差异的地方

- **支付调起**：H5 用 `WeixinJSBridge.invoke`，小程序用 `wx.requestPayment`；
- **登录方式**：H5 用手机号，小程序用 `wx.login` + 手机号授权；
- **原生能力**：小程序可调扫码、订阅消息；H5 需走公众号能力。

这些差异应在架构上隔离：后端逻辑统一，前端只在"最后一公里"按平台适配。

## 11. 小程序框架搭建后的下一步操作步骤

> 小程序基础框架 `/mini-program` 已搭建完成。在店铺端小程序 AppID、商户号、开放平台等资质到位前，仍有大量技术工作可以并行推进。本节把"下一步"变成可直接执行的操作步骤。

### 11.1 当前可立即推进的工作（不依赖微信资质）

| 工作项 | 依赖 | 优先级 | 说明 |
|--------|------|--------|------|
| 小程序首页接入真实店铺/服务数据 | 后端 `/api/shops/*` 接口 | P0 | 验证共享层是否工作，让小程序不再是空壳 |
| 小程序预约页复用 H5 预约逻辑 | `shared/lib/membership.ts`、后端预约接口 | P0 | 保证两端预约体验和数据一致 |

**理由：**

1. **不依赖微信资质**：不需要 AppID、商户号，直接复用现有后端接口；

3. **小程序从"空壳"变成"可用预览"**：店铺端可以先用微信开发者工具看到效果；
4. **为后续登录和支付打好基础**：页面数据流跑通后，加登录和支付只是增量改造。

### 11.3 具体操作步骤

#### 步骤 1：创建小程序端 API 模块

在 `mini-program/api/` 下创建：

- `shop.js`：获取店铺信息、服务列表、发型师列表；

- `customer.js`：获取当前顾客信息（先用 `customerId` 明文，后续统一改 token）。

这些模块内部调用 `mini-program/utils/api.js` 的 `get/post`，API 基础地址自动来自 `shared/api-base.ts`。

#### 步骤 2：小程序首页接入店铺数据

修改 `pages/index/index.js`：

1. `onLoad` 时调用 `shopApi.getShop('shop1')`；
2. 把店铺名称、地址、电话、营业时间渲染到首页；
3. 展示店铺服务列表入口。

**同步要求：**

- 首页展示的数据字段必须与 H5 首页一致；
- 服务列表的数据结构必须直接复用 `shared/types.ts` 中的 `Service` 类型。

#### 步骤 3：小程序预约页复用 H5 预约逻辑

修改 `pages/booking/booking.js`：

1. 加载默认店铺的服务列表和发型师列表；
2. 生成未来 7 天日期和可用时段；
3. 实现"随机发型师"自动匹配逻辑，**必须与 H5 的 `Booking.tsx` 算法一致**；
4. 点击"确认预约"时调用 `bookingApi.createBooking`，参数结构与 H5 保持一致。

**同步要求：**

- 预约页的数据结构、字段命名、状态值必须与 H5 相同；
- 随机发型师匹配算法必须复用 `shared/lib/` 中的逻辑（如果 H5 匹配逻辑还没抽到 shared，本次一并抽象）。

### 11.5 小程序 UI 统一改造规范

> 小程序与 H5 顾客端面向同一批用户，视觉与交互必须一致。本节把宪章 2.5.4 的原则落到小程序开发的具体文件和规范。

#### 设计 Token

- 唯一来源：`mini-program/app.scss` 中的 CSS 变量。
- 关键变量与 H5 对应关系：

| 小程序变量 | 取值 | H5 对应 |
|---|---|---|
| `--primary` | `#2563eb` | `blue-600` |
| `--accent` | `#f97316` | `orange-500` |
| `--success` | `#22c55e` | `green-500` |
| `--danger` | `#ef4444` | `red-500` |
| `--background` | `#f9fafb` | `gray-50` |
| `--text-primary` | `#1f2937` | `gray-800` |
| `--text-secondary` | `#6b7280` | `gray-500` |

- 扩展变量时，必须先在 `app.scss` 定义，并同步确认/新增 H5 端对应色值。

#### 通用组件

- **图标**：统一使用 `components/icon/icon`，页面引用名为 `mbs-icon`。所有页面 `.json` 必须在 `usingComponents` 中注册 `"mbs-icon": "/components/icon/icon"`。
- **按钮**：统一使用 `app.scss` 提供的 `.btn`、`.btn-primary`、`.btn-secondary`、`.btn-primary-plain`、`.btn-small`。
- **卡片**：统一使用 `.card`、`.card-title`、`.card-subtitle`。
- **标签/徽章**：统一使用 `.tag-*`、`.badge-*`、`.status-*`。
- **空状态**：统一使用 `.empty-state`、`.empty-icon`、`.empty-text`。
- **底部固定栏**：统一使用 `.fixed-bottom-bar`。
- **登录弹窗**：统一使用 `components/login-modal/login-modal`。

#### 页面结构模板

每个顾客端页面推荐按以下顺序组织 WXML：

1. `.gradient-header-*` 顶部渐变头部（含页面标题与副标题）；
2. `.container` 内容区，内部用 `.card` 堆叠；
3. 需要时底部 `.fixed-bottom-bar`；
4. 全局一次引用的 `<login-modal />`。

#### 新增/调整 UI 的检查清单

- [ ] 是否能在 H5 找到对应页面/组件？若结构不同，需说明原因。
- [ ] 颜色是否来自 `app.scss` 变量？是否同步 H5 颜色？
- [ ] 图标是否已在 `components/icon/icon.wxml` 中定义？名称是否与 H5 一致？
- [ ] 状态/文案是否与 H5 相同？是否复用 `shared/lib/membership.ts` 等共享层？

### 11.7 测试阶段技术调整

> 为简化微信开发者工具配置、避免 TypeScript 编译问题，当前 `/mini-program` 目录已统一使用 `.js` 文件运行。该调整仅影响测试阶段源码格式，不影响后期正式版功能；小程序仍与 H5/网页端共用同一套后端 API 和数据库，三方协同逻辑不变。

本地验证时请注意：

1. **导入目录**：在微信开发者工具中选择「导入项目」，路径指向仓库根目录下的 `mini-program` 子目录（不是整个仓库根目录）。
2. **关闭域名校验**：开发版小程序默认会校验 request 合法域名，测试环境请在「详情 → 本地设置」中勾选「不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书」，以允许请求未备案的测试域名。
3. **模拟登录态**：当前顾客端尚未接入 `wx.login`，请在微信开发者工具控制台执行：
   ```js
   wx.setStorageSync('mbs_customer_id', '真实顾客ID');
   ```
   之后重新加载首页/预约页，即可用该顾客身份调用后端接口。
4. **确认文件后缀**：`app.json` 中页面路径无需写后缀，但对应文件必须存在 `.js`（如 `pages/index/index.js`）。若仍残留 `.ts` 文件，需删除避免开发者工具混淆。
5. **API 响应解包**：后端统一返回 `{ success: true, data: ... }`，小程序端需在 `mini-program/api/*.js` 中解包 `result.data`，不能直接返回 `get/post` 的原始结果。
6. **避免使用 H5 特有 API**：
   - ❌ `URLSearchParams` → 改用手动 `encodeURIComponent` 拼接 query string；
   - ❌ `localStorage` → 改用 `wx.setStorageSync` / `wx.getStorageSync`；
   - ❌ `fetch` → 改用 `wx.request`；
   - ❌ WXML 中不支持 `.find()`、箭头函数、`?.` 可选链 → 在 `.js` 中预计算后绑定。
7. **网络超时**：Vercel Serverless 冷启动可能较慢，建议把 `wx.request` 超时设为 30 秒。
8. **使用生产域名测试**：Vercel 自动分配的 `*.vercel.app` 域名在国内可能访问超时，测试时直接使用已备案的生产域名（如 `www.hfmbs.cn/api`）。
