# MBS 理发店管理系统 — 后续开发计划书

## 当前状态

**演示版已经能用**：登录、客户增删改查、客户画像、CSV 导出。数据存在浏览器里，换设备会丢失。

---

## 总体目标

让理发店能真正用起来：员工各自登录 → 录入客户 → 数据永久保存 → 换设备也能看到。

---

## 分阶段执行

### 第一阶段：演示版跑稳（1-2 天）

**目标**：不追求数据持久化，先把所有功能在浏览器里跑通。

| 任务 | 做什么 | 验证方式 |
|------|--------|---------|
| 1.1 | 登录后能进 `/shop` 页面 | 输入手机号 13800000001，密码 123456 → 看到仪表盘 |
| 1.2 | 侧边栏有「客户表格管理」入口 | 左侧菜单出现新选项 → 点进去能看到 24 字段表格 |
| 1.3 | 客户管理有「添加客户」按钮 | 点客户管理 → 看到按钮 → 点击弹出表单 |
| 1.4 | 客户详情有「编辑信息」按钮 | 点任意客户 → 弹出详情 → 底部有编辑按钮 |
| 1.5 | 添加客户后刷新还在 | 添加 → 刷新 → 数据没丢 |
| 1.6 | CSV 导出能用 | 客户表格管理 → 点导出 → 下载 Excel |

**验收标准**：CEO、店长、客服、发型师 4 个角色都能登录，客户增删改查正常，刷新不丢数据。

---

### 第二阶段：接入数据库 Supabase（3-5 天）

**目标**：数据真正永久保存，换电脑/清缓存不丢。

#### 2.1 注册 Supabase 并建表（你自己做，不需要 agent）

1. 打开 [supabase.com](https://supabase.com)，用 GitHub 账号注册
2. 创建新项目，区域选 **Asia Pacific**（国内访问快）
3. 在 SQL Editor 中粘贴项目里的 `schema.sql` 并执行
4. 复制保存两个值：`SUPABASE_URL` 和 `SUPABASE_SERVICE_ROLE_KEY`

**验证**：Supabase 后台 Table Editor 能看到建好的表，能手动插入一行数据。

#### 2.2 恢复后端 API（约 2-3 次对话）

| 步骤 | 做什么 | 预计对话次数 |
|------|--------|------------|
| 2.2.1 | 恢复 `api/db/index.ts` Supabase 连接代码 | 1 次 |
| 2.2.2 | 恢复 `api/routes/auth.ts` 登录接口 | 1 次 |
| 2.2.3 | 恢复 `api/routes/customers.ts` 客户 CRUD 接口 | 1 次 |
| 2.2.4 | 在 Vercel 环境变量里添加 SUPABASE_URL 和 SUPABASE_KEY | 自己操作 |

**每次对话模板**：
```
我现在在做第 2.2.X 步
当前状态：[简述]
需要你做：[具体任务]
```

#### 2.3 前端切回真实 API（约 1-2 次对话）

| 步骤 | 做什么 |
|------|--------|
| 2.3.1 | `src/api.ts` 中 `USE_REAL_API` 改为 `true` |
| 2.3.2 | 删除 `CustomerManagement.tsx` 中本地 mock 读取逻辑 |
| 2.3.3 | 保留 localStorage 作为缓存（API 失败时回退） |

**验收标准**：添加客户 → 换浏览器登录 → 数据还在。任意设备登录都能看到同一份数据。

### 第三阶段：业务功能完善（持续迭代）

**目标**：围绕理发店实际运营需求逐步补功能。

| 优先级 | 功能 | 说明 |
|--------|------|------|
| 高 | 预约管理 | 客户预约、爽约提醒、排队叫号 |

## 工作方式规范

### 一次只改一处的操作流程

```
1. 告诉 agent 要改哪个文件、改成什么样
2. agent 修改 → npm run build 通过
3. git add → git commit → git push
4. 等 Vercel 部署（1-2 分钟）
5. 打开网页验证
6. 通过 → 改下一处；不通过 → 贴报错给 agent
```

### 每次对话开头的模板

```
我现在在做第 X 阶段第 Y 步
当前状态：[简述]
需要你做：[具体任务]
```

### 不要做的事

- 同时改前端和后端（混着改出问题难定位）
- 晚上 11 点后部署（头脑不清醒容易踩坑）
- 一次改多个文件（出问题不知道哪个导致）
- 不验证就改下一处（问题会累积）

## 回退方案

如果某一步搞砸了：

```powershell
git log --oneline -5        # 看最近 5 次提交
git reset --hard <commit>   # 回到某个正常的版本
git push --force            # 强制推送（谨慎使用）
```

建议每完成一个阶段在 GitHub 上打 tag：

```powershell
git tag v1.0-demo           # 演示版完成
git tag v2.0-supabase       # 数据库接入完成
git push --tags
```

## 踩坑预防清单（接 Supabase 前必读）

在进入第二阶段之前，记住以下预防措施：

1. **Base64 传文件**：如果 agent 给你一个长 Base64 字符串，直接 git push 让 Vercel 部署比在 PowerShell 里解码更可靠
2. **先 npm install**：删除任何文件后，先 `npm install` 再 `npm run build`
3. **相对路径数清楚**：`src/api.ts` 导入 `shared/` 是 `../shared/`，不是 `../../shared/`
4. **空数组是 truthy**：`if (result.data)` 不安全，用 `if (result.data?.token)`
5. **Zustand 两层同步**：改了 localStorage 记得同步 Zustand store
6. **Vite 代理不影响生产**：`vite.config.ts` 的 proxy 只在本地生效，不要担心影响 Vercel
7. **Vercel 12 函数限制**：免费版最多 12 个 Serverless Function，接 Supabase 后端时注意
8. **btoa 不支持中文**：永远用 `encodeURIComponent` 代替 `btoa`
